# 01 时间线记录（从需求到当前）

> 注：以下为本轮协作中的关键节点提炼，用于快速复盘。

## T1. 小程序建设目标确认
- 目标：基于 `MediCareAI` 现有后端，补齐微信小程序前端目录与适配。
- 约束：优先复用既有后端，不新增破坏性接口。

## T2. 基础功能打通
- 页面层：登录、注册、首页、症状提交流程、病历列表/详情、个人资料/慢病管理。
- API 层：完成认证、患者档案、病历、AI 诊断、文档上传、医生列表、疾病字典等调用链路。

## T3. 语法与配置问题处理
- 问题：真机调试报错 `app.json Unexpected token ... at position 0`。
- 根因：JSON 文件首字符 BOM（UTF-8 BOM）导致。
- 处理：统一无 BOM 编码后恢复。

## T4. 后端启动与联调位置澄清
- 用户在本地 Docker Desktop 已有实例。
- 协作中明确：小程序本身不在 Docker 内运行，使用微信开发者工具运行；后端在 Docker 服务中提供 API。

## T5. 登录疑问与平台说明
- 用户疑问：是否默认患者端，为何 Web 账号不能登录。
- 结论：小程序是患者端调用，需满足患者端账号/权限逻辑（`platform: patient` + `X-Platform: patient`）。

## T6. 真机调试与域名/地址问题
- 讨论点：未备案小程序能否本地真机调试、域名校验影响。
- 结论：本地调试可行，但真机不能使用 `127.0.0.1`，需局域网 IP；协议也要一致。

## T7. 全量中文化
- 需求：将页面都替换成中文。
- 范围：`app.json`、所有 `pages/*` 可见文案、`utils` 默认错误提示。
- 结果：页面文案、错误提示、空态、角色/状态/严重程度映射均中文化。

## T8. 首页“信息不全 + request:fail”问题
- 现象：首页角色/邮箱显示空，出现 `request:fail`。
- 关键定位：
  - 病历请求失败导致后续渲染受影响；
  - 请求协议可能被重定向成 `https://127.0.0.1...` 导致失败。
- 处理：
  - `casesApi` 改为 `/medical-cases/`；
  - 首页先渲染用户信息再拉病历；
  - `request:fail` 映射为可读中文错误。

## T9. 样式体验优化
- 全局样式重构：卡片、按钮、输入框、状态条、空态风格统一。
- 页面补充：辅助说明文案、进度提示、成功提示、空态引导。

## T10. 真机按钮不居中修复
- 现象：按钮文字偏移。
- 处理：按钮改用 `flex` 居中，替代纯 `line-height` 对齐。
